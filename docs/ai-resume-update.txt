二、项目经验 — 新增"AI 智能服务模块"（放在原项目经验后面）
========================================================================

项目：智慧社区 AI 智能服务模块
技术栈：Spring Boot + Spring AI + DeepSeek API + MySQL + Embedding + RAG

项目介绍：
在原有智慧社区微服务平台的基础上，独立设计并实现了一个 AI 智能服务模块（ai-service），
将大模型能力引入社区物业管理场景。整个模块涵盖智能客服问答、工单智能分析、运营数据
洞察与周报生成等四条业务链路，以及配套的知识库管理和 AI 调用可观测性体系。

1. 设计并实现了基于 RAG 的智能客服问答系统。采用关键字 + 向量语义的混合检索策略，
   先从 MySQL 知识库中检索与居民问题相关的社区资料（公告、维修电话、制度等），再将
   资料拼入 Prompt 交由 DeepSeek 生成回答。检索层设计了加权合并算法（关键字命中权重
   更高，向量辅助语义兜底），并加入了"无资料拒绝"机制，避免模型在缺乏依据时编造回答。

2. 实现工单智能分析功能。通过 System Prompt 约束模型输出结构化的工单分类结果
   （故障类别、优先级、风险等级、建议响应时间等），并在输出层设计了 Normalizer
   校验器：对模型返回的优先级做限幅校准，结合高危关键词（困人/起火/燃气泄漏等）
   和模糊词（不确定/好像等）做二次判断，降低误判风险。

3. 实现运营数据洞察与周报生成。支持手动指标传入和自动数据库聚合两种模式。模型
   基于社区报修、投诉、欠费等运营指标，自动生成风险洞察卡片和可执行行动项。同时
   设计了 Fallback 降级策略：当大模型不可用时，自动切换为基于业务规则的保底输出，
   确保运营页面不会出现空白。

4. 搭建了完整的知识库管理体系。支持知识文档的 CRUD 管理、向量 Embedding 重建
   （可切换本地 Hash 演示模式与 OpenAI 兼容的语义 Embedding），以及公告知识的
   定时同步任务。

5. 建立了 AI 调用可观测性机制。为每条 AI 调用链路设计了统一的日志记录模型
   （包含请求 ID、业务类型、调用耗时、置信度、成功/降级/失败状态），便于追踪
   每次大模型调用的成本和效果，也方便后续做质量分析和问题排查。

========================================================================
【附】AI 模块的关键技术点（面试准备用，不写进简历）
========================================================================

你可以根据这些点准备面试回答：

1. RAG 架构：检索 → 增强 → 生成 → 校验 四步流程
   - 为什么需要检索？（模型不知道本项目知识；避免编造）
   - 为什么用混合检索？（关键字精确匹配 + 向量语义兜底）
   - 检索完没找到资料怎么处理？（不调模型，直接返回兜底回答）

2. Prompt Engineering：
   - System Prompt 五要素（角色、边界、兜底、枚举、格式）
   - 为什么 system 放固定内容？（DeepSeek KV Cache，缓存命中便宜 50 倍）
   - User Prompt 的上下文块编号模式

3. Normalizer 模式：
   - 为什么不能直接信任模型输出？（非确定性：空字段、编造 ID、非法值）
   - 做了哪些校验？（白名单过滤、数字 clamp、枚举匹配、高危词二次判断）

4. Fallback 策略：
   - AI 调用失败时怎么处理？（规则生成保底内容，页面不空白）
   - 四条链路的不同降级策略

5. 可观测性：
   - 为什么需要？（成本追踪、质量监控、问题排查）
   - 记录了哪些字段？（bizType、status、confidence、latencyMs、retrievedSourceIds）

6. 技术选型：
   - 为什么用 Spring AI？（统一底层调用，直接调用 OpenAI 兼容接口）
   - 模型选型？（deepseek-v4-flash 日常，deepseek-v4-pro 强推理）
   - Embedding 怎么选？（开发阶段用 hash 本地方案，上线切 OpenAI 兼容）

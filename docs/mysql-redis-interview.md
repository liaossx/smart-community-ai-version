## 11. MySQL 的索引为什么用 B+ 树？

**回答**：
> B+ 树和二叉树、Hash 的区别。二叉树如果数据是递增插入的，树会退化成链表——查一个数得从根扫到底，跟全表扫没区别。Hash 等值查询 O(1) 快，但不支持范围查询——查"id 在 1 到 100 之间"，Hash 索引用不了。
>
> B+ 树解决了这两个问题。它能自平衡——数据怎么插入，树的高度不会差太多，查找永远是 O(log n)。而且非叶子节点只存索引不存数据，同样一个磁盘页能存更多的索引键，树就更矮，IO 次数更少。叶子节点之间用双向链表串着，范围查询 id BETWEEN 1 AND 100 时，找到 1 之后顺着链表往后扫就行，不用回上层节点。

## 12. 什么是回表？你的车位余量查询用到索引覆盖了吗？

**当前代码**：[parking.sql](/D:/CCproject/smart-community-ai-version/docs/sql/parking.sql) 第 5-20 行

```sql
-- 当前 biz_parking_space 的索引（4 个单列索引）
INDEX idx_community_id (community_id),
INDEX idx_space_no (space_no),
INDEX idx_status (status),
INDEX idx_bind_user_id (bind_user_id)
```

```sql
-- 车位余量查询执行的 SQL
SELECT COUNT(*) FROM biz_parking_space
WHERE community_id = 1 AND space_type = 'TEMP' AND status = 'AVAILABLE'
```

**回表是什么**：MySQL 先用 idx_community_id 索引找到 community_id=1 的所有行——但索引里只有 community_id 和主键 id，没有 space_type 和 status。所以 MySQL 拿到主键 id 后，要再回主键索引（聚簇索引）去查这一行的完整数据，拿到 space_type 和 status 才能判断。**这个"回主键索引再查一次"就叫回表。**

**当前用到索引覆盖了吗**：**没有。** 四个单列索引，没有一个同时覆盖 community_id、space_type、status。MySQL 用 idx_community_id 过滤完后，每一行都要回表。

**如果要做索引覆盖，怎么改**：

```sql
-- 建一个联合索引，三个查询字段全在索引里，不用回表
ALTER TABLE biz_parking_space ADD INDEX idx_cov_remain (community_id, space_type, status);
```

有了这个索引后，MySQL 直接在索引里就能拿到 community_id + space_type + status，COUNT(*) 全程在索引页里完成，一次回表都不用。不过开发阶段没加这个索引——缓存已经把它从 292ms 降到 3ms 了，再加索引优化的边际收益很小。

## 13. MySQL 的事务隔离级别？默认级别下的问题？怎么解决幻读？

**回答**：
> 四种隔离级别：读未提交、读已提交、可重复读、串行化。InnoDB 默认是可重复读。
>
> 可重复读解决的问题是"同一个事务里两次读到的数据不一样"——通过 MVCC（多版本并发控制），每个事务看到的是自己开始时的数据快照，别人改了也看不到。
>
> 可重复读下还是可能有幻读——比如事务 A 查"社区 1 的车位总数"是 100 个，事务 B 同时插入了一个新车位记录，事务 A 再查变成了 101 个——多出来一行，像幻觉一样。解决方式：MVCC 的快照读避免了一部分，但当前读（SELECT ... FOR UPDATE）会有幻读风险。InnoDB 用 Next-key Lock 解决——它锁住的不是单独一行，而是行 + 它前面的间隙。事务 B 想插进这个间隙就会被阻塞，等到事务 A 提交后才插入。
>
> 我项目里事务隔离级别是默认的可重复读，没改过。报名接口里用的 Redis + MQ 来控并发，数据库层面靠唯一索引防重复。没有特别用 FOR UPDATE 去锁行，因为 Redis 预扣这一步已经把并发扛住了。

## 14. Redis 除了缓存还做过什么？Redis 挂了怎么保证数据不丢？

**回答**：
> 我项目里 Redis 做了三件事：缓存（车位余量、房屋信息）、分布式锁（停车出闸幂等、报名库存预扣）、Set 防重复报名。不只是缓存。
>
> 持久化靠两种机制：RDB 和 AOF。RDB 是定时快照——Redis 每隔一段时间把整个内存数据写成一个二进制的 dump 文件，恢复的时候直接加载，快但两次快照之间如果挂了会丢数据。AOF 是写日志——每收到一个写命令就追加到日志文件，数据更安全，但文件更大、恢复更慢。
>
> 生产环境一般两个都开——RDB 做主备份，AOF 做增量保证不丢太多数据。Redis 挂了重启后，先加载 RDB 恢复到最近一次快照，再重放 AOF 补上快照之后到挂掉之间的数据。我项目里暂时没遇到持久化需求——分布式锁的数据丢了也不会出大问题，最多当时正在出闸的请求被拒绝，用户重试即可。
>
> 如果面试官追问"Redis 主从 + 哨兵"：单个 Redis 挂了还有从节点顶上，哨兵自动做主从切换。

## 15. 缓存穿透和缓存击穿怎么解决？布隆过滤器原理？

**回答**：
> 穿透——有人用不存在的 ID 查数据，缓存和数据库都没有。我拿房屋详情接口举例：GET /api/house/{houseId}，houseId 是用户传的，可以遍历假 ID 打库。当前项目里查出来 null 时不缓存（disableCachingNullValues），但攻击者用大量假 ID 扫描，还是会每次都穿透到 MySQL。解决办法有两种：轻量的用缓存空对象——查出来 null 也写缓存，设短过期时间 30 秒。重的用布隆过滤器——在缓存前面放一个过滤器，把所有合法的 houseId 预先存进去。请求来了先过布隆，不在集合里直接返回 404，连缓存都不用查。
>
> 击穿——热点 key 过期了，大量请求同时打数据库。我项目用定时预热解决：每 5 分钟主动把所有社区的车位余量缓存写入 Redis，用户打开页面的第一次查询也是缓存命中 3ms。
>
> 布隆过滤器的原理：不存数据本身，用多个 Hash 函数把一个值映射到一个二进制数组的多个位上。查的时候看这几个位是不是都是 1——都是 1 说明"可能存在"，有一个是 0 就"一定不存在"。它不存数据所以内存占用极小，但有误判率（把不存在的误判为存在）。

---

**提问**：如果面试官追问"布隆过滤器删除数据怎么办，车位不是可能被删吗"，你回答：
> 布隆过滤器设置了就不能删位。业务上房屋信息很少物理删除——用户端看不到的房源用软删除，把 status 设成 DELETED，数据库里 row 还在，布隆过滤器不需要删。
